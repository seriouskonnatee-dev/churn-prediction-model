# Model Card — Churn Prediction Model

## Intended use

Ranks active subscription/retail customers by estimated probability of churning in the next billing
period, so a retention team can prioritize proactive outreach. Intended as a **decision-support** input
for a human-reviewed retention workflow, not an automated action trigger.

## Models

| Model | Type | Notes |
|---|---|---|
| Logistic Regression | Linear, interpretable baseline | `class_weight="balanced"`, scaled numeric features |
| Random Forest | Non-linear ensemble | 300 trees, `max_depth=8`, `class_weight="balanced"` |

## Training data

5,000 synthetic customer snapshots generated by `src/data_generation.py` (see `docs/design.md` for the
full data dictionary and generating process). Churn rate ~26%, reflecting a realistic imbalanced-classes
subscription-business scenario. **This is synthetic data for portfolio purposes — no real customer
records were used or are represented.**

## Evaluation

75/25 stratified train/test split. See `outputs/model_comparison.csv` and `notebooks/churn_prediction.ipynb`
for the exact run; at last execution:

| Model | ROC-AUC | Precision | Recall | F1 |
|---|---|---|---|---|
| Logistic Regression | ~0.77 | ~0.28 | ~0.72 | ~0.41 |
| Random Forest | ~0.75 | ~0.31 | ~0.57 | ~0.40 |

Precision is low at the default 0.5 threshold because `class_weight="balanced"` intentionally biases
both models toward recall (catching more true churners) at the cost of more false positives — a
reasonable trade-off for a retention-outreach use case where a missed churner is more costly than an
unnecessary check-in call, but the threshold should be tuned to the retention team's actual cost/benefit
ratio before deployment, not left at 0.5.

## Limitations

- **Synthetic data.** Feature-churn relationships were hand-specified to be plausible, not fit to real
  behavior — absolute metrics will not transfer to a real customer base; only the *methodology*
  (feature engineering, model comparison, evaluation approach) is the portfolio artifact here.
- **No temporal validation.** All rows are independent synthetic snapshots; a production version should
  validate on a genuinely held-out future time period, not just a random split, to catch concept drift.
- **Correlation, not causation.** Feature importance shows association with churn, not the causal effect
  of an intervention — e.g. "customers without an add-on churn more" does not mean *selling* an add-on to
  a specific customer will reduce their churn risk by the same amount. Uplift modeling would be needed
  for that question.
- **Class imbalance and threshold sensitivity.** Reported precision/recall are threshold-dependent
  (0.5 cutoff); see the notebook's ROC curve for the full precision/recall trade-off space before picking
  an operating threshold.
- **Not validated for fairness.** No audit was performed for disparate impact across customer segments;
  before any real deployment this should be checked, particularly since `payment_method` can correlate
  with socioeconomic factors.
